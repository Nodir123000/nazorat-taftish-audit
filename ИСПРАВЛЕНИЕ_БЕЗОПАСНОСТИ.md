### [C-1] Фиктивный пользователь-администратор в `getCurrentUser()`

**Файл:** `lib/auth.ts`  
**Серьёзность:** КРИТИЧЕСКАЯ  
**CVSS:** ~9.8 (несанкционированный полный доступ)

**Проблема (до исправления):**
```typescript
// УЯЗВИМЫЙ КОД — если куки нет, возвращался фиктивный admin!
if (authUser) {
  return JSON.parse(authUser.value) as User
}
return {
  user_id: 1,
  username: "admin",
  role: "admin",          // ← ПОЛНЫЙ ДОСТУП БЕЗ АУТЕНТИФИКАЦИИ
  is_active: true,
  ...
}
```

Любой пользователь, не имеющий куки сессии (в том числе внешний злоумышленник), получал объект пользователя с ролью `admin`. Все защищённые страницы и API-маршруты были полностью открыты.

**Исправление:**
```typescript
// БЕЗОПАСНЫЙ КОД — возвращаем null если сессии нет
if (!authCookie?.value) {
  return null           // ← пользователь перенаправляется на /login
}
```

Функция теперь возвращает `null` при отсутствии сессии, что корректно обрабатывается в layout-компонентах (`redirect("/login")`).

### [C-2] Подделка куки сессии (Cookie Tampering)

**Файл:** `lib/auth.ts`  
**Серьёзность:** КРИТИЧЕСКАЯ  
**CVSS:** ~8.8 (повышение привилегий)

**Проблема (до исправления):**
```typescript
// УЯЗВИМЫЙ КОД — простой JSON без проверки целостности
cookieStore.set("auth_user", JSON.stringify(user), { ... })

// Получение — просто парсим без проверки подписи
return JSON.parse(authUser.value) as User  // ← можно подделать роль!
```

Пользователь мог вручную изменить куки в браузере (DevTools → Application → Cookies), изменив `role` с `"inspector"` на `"admin"`, и получить полный доступ.

**Исправление — HMAC-SHA256 подпись куки:**
```typescript
// Подписываем данные: encoded.HMAC_SHA256(encoded, JWT_SECRET)
function signData(data: string): string {
  const encoded = Buffer.from(data).toString("base64url")
  const hmac = createHmac("sha256", getSecret())
  hmac.update(encoded)
  return `${encoded}.${hmac.digest("base64url")}`
}

// Проверяем подпись перед использованием (constant-time сравнение)
function verifyAndDecode(signed: string): string | null {
  // ...timingSafeEqual() предотвращает timing-атаки
  if (!timingSafeEqual(sigBuf, expBuf)) return null
  return Buffer.from(encoded, "base64url").toString("utf-8")
}
```

Теперь любое изменение куки делает подпись недействительной — сессия сбрасывается.

### [C-3] Отсутствие middleware аутентификации

**Файл:** `middleware.ts` (создан новый)  
**Серьёзность:** КРИТИЧЕСКАЯ  
**CVSS:** ~9.1 (обход контроля доступа)

**Проблема (до исправления):**  
Не существовало глобальной защиты маршрутов. Каждый API-маршрут проверял аутентификацию самостоятельно (или не проверял вовсе). При наличии `getCurrentUser()` с фиктивным пользователем защита была полностью иллюзорной.

**Исправление — Next.js Edge Middleware:**
```typescript
// middleware.ts (Edge Runtime)
export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl

  if (isPublicPath(pathname)) return NextResponse.next()

  const isAuthenticated = await isValidSession(authCookie?.value)

  if (!isAuthenticated) {
    if (pathname.startsWith("/api/")) {
      return NextResponse.json({ error: "Не авторизован" }, { status: 401 })
    }
    return NextResponse.redirect(new URL("/login", request.url))
  }
}
```

Middleware перехватывает ВСЕ запросы до попадания в роут-хендлеры:
- API-маршруты: возвращает `401 JSON`
- Страницы: перенаправляет на `/login`
- Проверяет HMAC-подпись куки через Web Crypto API (Edge-совместимо)

### [C-4] Слабый секрет JWT

**Файл:** `.env`  
**Серьёзность:** КРИТИЧЕСКАЯ  
**CVSS:** ~9.1 (полная компрометация подписей)

**Проблема:**
```
JWT_SECRET="your-secret-key"  ← предсказуемый, 14 символов
```

**Исправление:**
```
JWT_SECRET="38656724e8b36b6828d3d61cd22ee858d8e00a0d8b4aa860af66a81491e..."
```
Сгенерирован криптографически стойкий секрет (96 hex-символов = 384 бита) командой:
```bash
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
```

Код проверяет минимальную длину в production:
```typescript
function getSecret(): string {
  const secret = process.env.JWT_SECRET
  if (!secret || secret === "your-secret-key" || secret.length < 32) {
    if (process.env.NODE_ENV === "production") {
      throw new Error("JWT_SECRET должен быть ≥ 32 символов!")
    }
  }
}
```

## Архитектура защиты (после исправлений)

```
HTTP-запрос
    ↓
┌─────────────────────────────────┐
│     middleware.ts (Edge)         │
│  • Проверка: есть ли куки?       │
│  • Проверка HMAC-подписи куки    │
│  • /api/* → 401 JSON             │
│  • /pages/* → redirect /login    │
└────────────────┬────────────────┘
                 ↓ (если аутентифицирован)
┌─────────────────────────────────┐
│  Layout / Page (Server Component)│
│  getCurrentUser()               │
│  • Верифицирует HMAC            │
│  • Проверяет user в БД (is_active)│
│  • Возвращает User или null      │
└────────────────┬────────────────┘
                 ↓
┌─────────────────────────────────┐
│    API Route Handler            │
│    Бизнес-логика                │
└─────────────────────────────────┘
```

**Двойная защита:** middleware (Edge) + server-side DB validation. Даже если одна из защит будет обойдена, вторая задержит злоумышленника.

## Что ещё рекомендуется сделать (вне данного PR)

### Среднесрочно (1–2 месяца)
1. **Redis для rate limiting** — текущий in-memory ограничитель сбрасывается при перезапуске. Для production использовать `@upstash/ratelimit` или `ioredis`.
2. **Шифрование ПИНФЛ и паспортных данных** — данные хранятся в открытом виде в БД. Применить AES-256 на уровне Prisma middleware.
3. **Nonce-based CSP** — заменить `unsafe-inline` на криптографические nonce для каждого запроса.
4. **Аудит неудачных входов** — добавить запись в `audit_log` для брутфорс-мониторинга.
5. **Ротация секрета JWT** — реализовать смену ключа без разлогинивания всех пользователей.

### Долгосрочно (квартал)
6. **Тесты безопасности** — Playwright E2E тесты для login flow, auth bypass попыток.
7. **Сканирование зависимостей** — `npm audit` в CI/CD при каждом merge.
8. **Penetration testing** — внешний аудит перед вводом в эксплуатацию.
9. **SIEM интеграция** — подключить audit_log к системе мониторинга событий.
10. **MFA** — двухфакторная аутентификация для ролей `admin` и `chief_inspector`.
