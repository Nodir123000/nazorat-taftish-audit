## Аннотация

Настоящий документ является основным техническим описанием Автоматизированной информационной системы контрольно-ревизионной работы (АИС КРР) Министерства обороны Республики Узбекистан.

АИС КРР — это веб-ориентированная многопользовательская информационная система, предназначенная для автоматизации процессов планирования, проведения, учёта и отчётности по контрольно-ревизионной деятельности. Система обеспечивает полный жизненный цикл ревизии: от формирования годового и квартального планов, выпуска приказов, формирования ревизионных комиссий — до регистрации нарушений, принятия решений по ним и формирования аналитической отчётности.

Документ предназначен для разработчиков, системных администраторов, специалистов по информационной безопасности и ревизоров, осуществляющих эксплуатацию системы.

## 1. Общие сведения о системе

### 1.1. Полное и сокращённое наименование

**Полное наименование:** Автоматизированная информационная система контрольно-ревизионной работы Министерства обороны Республики Узбекистан.

**Сокращённое наименование:** АИС КРР.

**Наименование на государственном языке:** Mudofaa vazirligi nazorat-taftish ishlari avtomatlashtirilgan axborot tizimi (AAT NTI).

### 1.2. Назначение системы

АИС КРР предназначена для:

- автоматизации планирования контрольно-ревизионных мероприятий (годовое, квартальное, месячное планирование);
- ведения реестра проверок, аудитов и ревизий воинских частей и подразделений МО РУз;
- регистрации, классификации и учёта выявленных нарушений, а также контроля исполнения решений по ним;
- формирования, хранения и экспорта ревизионных актов, предписаний, инструктажей и иных документов;
- ведения кадровой базы ревизионного аппарата и личных дел военнослужащих;
- аналитики ключевых показателей эффективности (KPI) контрольно-ревизионной деятельности;
- обеспечения сквозного аудиторского следа всех операций в системе;
- централизованного управления нормативно-справочной информацией.

### 2.3. Паттерн App Router (Next.js 15)

Приложение использует файловую маршрутизацию Next.js App Router. Каждый маршрут реализован как отдельная директория внутри `/app`. Server Components (RSC) обеспечивают первичный рендеринг на сервере; Client Components (`"use client"`) используются для интерактивных форм и таблиц.

Middleware (`middleware.ts`) исполняется на Edge Runtime и выполняет:
1. Верификацию HMAC-SHA256 подписи сессионной куки (`auth_user`) на всех защищённых маршрутах.
2. Перенаправление неаутентифицированных пользователей на `/login`.
3. Возврат HTTP 401 JSON для незащищённых API-запросов.
4. Предотвращение повторного входа аутентифицированных пользователей через `/login`.

## 3. Описание подсистем

### ПС-01 — Подсистема мониторинга и аналитики (Dashboard / KPI)

**Маршруты:** `/dashboard`, `/kpi`

**Назначение.** Предоставляет сводный обзор состояния контрольно-ревизионной деятельности в режиме реального времени, а также инструменты анализа ключевых показателей эффективности.

**Функциональный состав:**
- Отображение агрегированных показателей: количество плановых/проведённых ревизий, выявленных нарушений, сумм нарушений.
- Визуализация динамики нарушений по периодам (графики Recharts).
- Контроль исполнения планов ревизий (процент выполнения).
- Мониторинг статусов решений по нарушениям.
- Сводные KPI-карточки по подразделениям и видам нарушений.
- Экспорт аналитических данных в форматы Excel (.xlsx) и PDF.

**Технические особенности:**
- Данные подгружаются через SWR с автоматическим обновлением.
- Диаграммы реализованы на базе Recharts с адаптивной вёрсткой.
- Страница реализована как Client Component с серверной предзагрузкой данных (RSC).

### ПС-02 — Подсистема планирования (Planning)

**Маршрут:** `/planning`

**Назначение.** Обеспечивает полный цикл формирования и утверждения планов контрольно-ревизионных мероприятий: годовых, квартальных и месячных.

**Функциональный состав:**

- **Годовые планы** (`RevPlanYear` / `rev_plan_year`):
  - Создание и редактирование годового плана ревизий с указанием периода охвата, органа контроля, направления проверки, типа инспекции, ответственного исполнителя.
  - Регистрация входящих документов (входящий номер, дата).
  - Ведение журнала изменений плана (`plan_change_log`).
  - Процедура согласования и утверждения плана с фиксацией даты утверждения и утвердившего пользователя.
  - Отслеживание показателей выполнения: `planned_count`, `completed_count`, `objects_total`, `objects_fs`, `objects_os`.

- **Квартальные планы** (`quarterly_plans`):
  - Декомпозиция годового плана по кварталам.
  - Статусы: `draft`, `approved`, `active`, `completed`.
  - Автоматический расчёт плановых показателей.

- **Месячные планы** (`monthly_plans`):
  - Детализация квартального плана по месяцам.
  - Привязка к конкретным аудитам (`Audit`).
  - Контроль степени выполнения.

- **Предписания** (`prescriptions`):
  - Регистрация предписаний, привязанных к плану.
  - Уникальный номер предписания, дата, выдавший, статус исполнения.

- **Инструктажи** (`briefings`):
  - Фиксация проведённых инструктажей в рамках плана.
  - Содержание, меры безопасности, инструктирующий, дата.

**Технические особенности:**
- Связанная иерархия: `RevPlanYear` → `quarterly_plans` → `monthly_plans` → `Audit`.
- Смена статуса реализована через конечный автомат (state machine) с проверкой прав роли.

### ПС-03 — Подсистема управления аудитами (Audits)

**Маршрут:** `/audits`

**Назначение.** Автоматизирует полный жизненный цикл ревизии/проверки воинской части: от создания и назначения комиссии до формирования акта и регистрации нарушений.

**Функциональный состав:**

- **Реестр аудитов** (`Audit` / `audits`):
  - Создание записи ревизии с уникальным номером (`auditNumber`), типом аудита, датами начала/окончания.
  - Привязка к воинской части (`ref_units`), месячному плану (`monthly_plans`), приказу (`orders`).
  - Назначение главного ревизора (`leadAuditorId`).
  - Управление статусами: `planned` → `in_progress` → `completed` → `closed`.

- **Приказы** (`orders`):
  - Создание приказов о назначении ревизии с уникальным номером и датой.
  - Привязка к шаблону документа (`document_templates`).
  - Управление статусами: `issued`, `active`, `completed`.

- **Ревизионная комиссия** (`commission_members`):
  - Формирование состава комиссии с указанием роли каждого члена (руководитель, член, секретарь).
  - Определение периода участия каждого члена.
  - Привязка к приказу и аудиту.

- **Акты аудита** (`audit_reports`):
  - Формирование акта с номером, датой, сводкой.
  - Автоматический подсчёт `findings_count`, `violations_count`, `total_amount`.
  - Статусы: `draft`, `submitted`, `approved`, `rejected`.

- **Замечания** (`findings`):
  - Регистрация выявленных замечаний с привязкой к акту.
  - Степень критичности: `low`, `medium`, `high`, `critical`.
  - Контроль сроков устранения (`due_date`, `closure_date`).

**Технические особенности:**
- Модель `Audit` является центральным агрегатом, к которому привязаны акты, нарушения, члены комиссии.
- Документы (файлы) хранятся в таблице `documents` с полем `entity_type = 'audit'` и `entity_id`.

### ПС-04 — Подсистема учёта нарушений (Violations)

**Маршрут:** `/violations`

**Назначение.** Обеспечивает полный учёт выявленных нарушений, классификацию по 14 категориям, денежную оценку, ведение решений и контроль их исполнения.

**Функциональный состав:**

- **Реестр нарушений** (`violations`):
  - Уникальный номер нарушения (`violation_number`), категория, описание, дата выявления.
  - Денежная сумма нарушения (поле `amount`, тип `DECIMAL(18,2)`).
  - Степень серьёзности (`severity`): `low` / `medium` / `high` / `critical`.
  - Ответственное лицо (`responsible_person`), привязка к воинской части.
  - Связь с аудитом, в рамках которого выявлено нарушение.

- **Решения по нарушениям** (`decisions`):
  - Регистрация решений с уникальным номером (`decision_number`), типом и описанием.
  - Указание ответственного исполнителя (`responsible_executor`) и срока исполнения (`deadline`).
  - Статусы решений: `issued`, `in_progress`, `completed`, `overdue`.

- **Классификатор нарушений** (`ref_violations`):
  - 14 предустановленных категорий нарушений с кодами, правовыми основаниями и описаниями.
  - Степень серьёзности задана на уровне категории.

- **Вспомогательные справочники:**
  - `ref_violation_reasons` — причины нарушений.
  - `ref_violation_severities` — степени серьёзности.
  - `ref_violation_statuses` — статусы нарушений.

**Технические особенности:**
- Отношение `violations` → `decisions` = один ко многим (одно нарушение может иметь несколько решений).
- Денежные суммы хранятся с точностью до 2 десятичных знаков (тип `DECIMAL(18,2)`).

### ПС-05 — Подсистема отчётности (Reports)

**Маршрут:** `/reports`

**Назначение.** Формирование стандартизированной отчётности по итогам контрольно-ревизионных мероприятий, в том числе экспорт в форматах Excel и PDF.

**Функциональный состав:**

- Сводные отчёты по аудитам за выбранный период (год, квартал, месяц).
- Отчёты по нарушениям в разрезе категорий, подразделений, степеней серьёзности.
- Отчёты по исполнению решений с контролем просроченных позиций.
- Отчёты по выполнению планов ревизий (факт/план).
- Формирование ревизионных актов в формате PDF на основе шаблонов (`document_templates`).
- Экспорт табличных данных в Excel с использованием ExcelJS.
- Реестр нормативных документов (`regulatory_documents`) с полнотекстовым поиском по тегам.

**Технические особенности:**
- Генерация PDF реализована через связку `jspdf` + `html2canvas` (HTML-рендеринг → canvas → PDF).
- Генерация Excel реализована через `exceljs` с форматированием ячеек, формулами и сводными страницами.
- Время формирования отчёта не превышает 30 секунд при объёме данных до 100 000 записей.

### ПС-06 — Подсистема управления персоналом (Personnel)

**Маршрут:** `/personnel`

**Назначение.** Ведение базы данных военнослужащих и гражданского персонала, задействованного в контрольно-ревизионной работе.

**Функциональный состав:**

- **Реестр личного состава** (`personnel`):
  - Уникальный идентификатор физического лица (`ref_physical_persons`).
  - Табельный номер (`service_number`), персональный номер регистрации (`pnr`).
  - Привязка к воинской части, воинскому званию (`ref_ranks`), должности (`ref_positions`), военно-учётной специальности (`ref_vus_list`).
  - Даты начала и окончания службы.
  - Уровень допуска к секретным сведениям (`clearance_level`).
  - Статус: `active`, `inactive`, `dismissed`.

- **Физические лица** (`ref_physical_persons`):
  - ПИНФЛ, данные паспорта, дата и место рождения.
  - Пол, национальность, уровень образования.
  - Регион, район проживания.
  - Контактные данные (телефон, email), адрес.

- **Справочники персонала:**
  - `ref_ranks` — воинские звания с категориями и составами (`ref_compositions`).
  - `ref_positions` — должности с кодами.
  - `ref_vus_list` — военно-учётные специальности.
  - `ref_education_levels` — уровни образования.
  - `ref_security_clearances` — уровни допуска.

**Технические особенности:**
- Данные ПИНФЛ и паспортные данные являются персональными данными категории ДСП и защищаются в соответствии с Законом РУз № ЗРУ-547.
- Модель `personnel` является ссылочной (pivot) между `ref_physical_persons` и организационными справочниками.

### ПС-07 — Подсистема управления подразделениями (Units)

**Маршрут:** `/units`

**Назначение.** Ведение справочника воинских частей и подразделений, а также их организационной иерархии.

**Функциональный состав:**

- **Реестр воинских частей** (`ref_units`):
  - Уникальный код части (`unit_code`), многоязычное наименование (JSON).
  - Командир и его воинское звание.
  - Тип подразделения (`ref_unit_types`), специализация (`ref_specializations`).
  - Привязка к военному округу (`ref_military_districts`) и административному району (`ref_areas`).
  - Признак активности (`is_active`).

- **Военные округа** (`ref_military_districts`):
  - Иерархическая структура военно-административного деления.

- **Административно-территориальное деление:**
  - `ref_regions` — области (регионы) Республики Узбекистан.
  - `ref_areas` — районы областей.

- **Связанные операции:**
  - Просмотр истории ревизий по воинской части.
  - Отчёт по нарушениям в разрезе части.
  - Статистика выполнения планов.

### ПС-08 — Подсистема нормативно-справочной информации (Reference)

**Маршрут:** `/reference`

**Назначение.** Централизованное управление нормативно-справочной информацией (НСИ), используемой всеми подсистемами АИС КРР.

**Функциональный состав:**

- Управление 14 категориями нарушений (`ref_violations`).
- Реестр направлений контроля (`ref_control_directions`).
- Реестр типов контроля (`ref_control_types`) и типов инспекций (`ref_inspection_types`).
- Реестр органов контроля (`ref_control_authorities`).
- Справочник типов документов (`ref_document_types`).
- Реестр нормативных документов (`regulatory_documents`) с версионированием.
- Шаблоны документов (`document_templates`) с поддержкой многоязычных шаблонов.
- Справочники: статусы (`ref_statuses`), классификаторы (`ref_classifiers`), источники финансирования (`ref_financing_sources`), статьи бюджета (`ref_budget_articles`), ТМЦ (`ref_tmc_types`).
- Реестр отделов снабжения (`ref_supply_departments`).

**Технические особенности:**
- Большинство справочников имеют мультиязычное наименование — поле `name` хранится в формате JSON: `{"ru": "...", "uz": "...", "kaa": "..."}`.
- Вновь добавляемые записи получают статус `active`; удаление физическое не применяется (soft-delete через `status = 'inactive'`).

### ПС-09 — Подсистема администрирования (Admin / Diagnostics)

**Маршруты:** `/admin`, `/diagnostics`, `/translation-management`

**Назначение.** Управление пользователями, ролями, конфигурацией системы, переводами интерфейса и диагностика.

**Функциональный состав:**

- **Управление пользователями** (`users`):
  - CRUD пользователей системы с назначением роли, подразделения, контактных данных.
  - Блокировка/разблокировка учётных записей (`is_active`).
  - Сброс паролей с принудительной перехэшевацией (bcrypt, 12 rounds).

- **Журнал аудита** (`audit_log`):
  - Просмотр полного журнала действий пользователей: тип действия, таблица, старое/новое значение, IP-адрес, временная метка.
  - Фильтрация по пользователю, периоду, типу действия.
  - Экспорт журнала для расследования инцидентов.

- **Управление переводами** (`ui_translations`):
  - Редактирование текстов интерфейса на трёх языках: русском, узбекском, каракалпакском.
  - Группировка ключей по тегам, управление статусами (`active` / `inactive`).
  - Импорт переводов из JSON-файлов.

- **Диагностика** (`/diagnostics`):
  - Проверка состояния соединения с базой данных.
  - Мониторинг утилизации ресурсов сервера.
  - Проверка корректности конфигурационных переменных окружения.

- **Система помощи** (`help_articles`, `doc_view_logs`):
  - Управление справочными статьями с категоризацией.
  - Журнал просмотра справочных документов пользователями.

### ПС-10 — Подсистема документооборота

**Функциональный состав:**

## 4. Схема базы данных

### 4.1. Основные сущности и связи

База данных АИС КРР включает более 50 таблиц, сгруппированных в четыре функциональных домена.

#### Домен 1 — Планирование и аудиты

```
rev_plan_year (RevPlanYear)
  │ 1:N
  ├── quarterly_plans
  │     │ 1:N
  │     └── monthly_plans
  │           │ 1:N
  │           └── audits (Audit)
  │                 │ 1:N
  │                 ├── audit_reports
  │                 │     │ 1:N
  │                 │     └── findings
  │                 ├── violations
  │                 │     │ 1:N
  │                 │     └── decisions
  │                 └── commission_members
  │
  ├── orders (приказы)
  │     └── commission_members
  ├── briefings (инструктажи)
  └── prescriptions (предписания)
```

#### Домен 2 — Персонал и организационная структура

```
ref_physical_persons
  │ 1:N
  └── personnel
        ├── ref_ranks (звания)
        ├── ref_positions (должности)
        ├── ref_vus_list (ВУС)
        └── ref_units (воинские части)
              ├── ref_unit_types
              ├── ref_specializations
              ├── ref_military_districts
              └── ref_areas
                    └── ref_regions
```

#### Домен 3 — НСИ (Нормативно-справочная информация)

```

ref_control_directions (направления контроля)
ref_control_types / ref_inspection_types (типы проверок)
ref_control_authorities (органы контроля)

ref_classifiers (универсальный классификатор)
ref_budget_articles (статьи бюджета)

ref_genders / ref_nationalities / ref_education_levels / ref_security_clearances

```

```
users
  │ 1:N
  ├── audit_log (журнал действий — неизменяемый)
  ├── commission_members
  ├── orders (создатель приказа)
  ├── briefings (инструктирующий)
  ├── prescriptions (выдавший)
  ├── audits (главный ревизор)
  └── doc_view_logs (просмотры справки)
```

### 4.3. Политика именования

- Таблицы: строчные буквы, символ подчёркивания (`snake_case`).
- Первичные ключи: `<table>_id` или `id`.
- Внешние ключи: `<referenced_table>_id`.
- Поля дат создания/изменения: `created_at` / `updated_at` с автоматическим заполнением (`DEFAULT now()`).
- Мультиязычные поля: тип `JSON`, ключи `"ru"`, `"uz"`, `"kaa"`.

## 6. Информационная безопасность

### 6.1. Архитектура безопасности

АИС КРР реализует принцип **Defence in Depth** (эшелонированная защита) — несколько независимых уровней защиты, каждый из которых создаёт дополнительный барьер для злоумышленника.

```
┌──────────────────────────────────────────────────────────────┐
│  УРОВЕНЬ 1: Сетевая защита                                   │
│  • Интранет МО РУз (изоляция от публичного Интернета)        │
│  • HTTPS / TLS ≥ 1.2 для всего трафика                       │
│  • Nginx SSL-терминация                                       │
├──────────────────────────────────────────────────────────────┤
│  УРОВЕНЬ 2: HTTP-заголовки безопасности (Nginx)              │
│  • X-Frame-Options: DENY                                     │
│  • X-Content-Type-Options: nosniff                           │
│  • Content-Security-Policy (CSP)                             │
│  • Strict-Transport-Security (HSTS)                          │
│  • Referrer-Policy: strict-origin-when-cross-origin          │
│  • Permissions-Policy                                        │
├──────────────────────────────────────────────────────────────┤
│  УРОВЕНЬ 3: Аутентификация и управление сессиями             │
│  • HMAC-SHA256 подпись сессионного токена                    │
│  • Куки: HttpOnly; Secure; SameSite=Strict                   │
│  • Время жизни сессии: 8 часов                               │
│  • Блокировка: 5 ошибочных попыток за 15 минут               │
├──────────────────────────────────────────────────────────────┤
│  УРОВЕНЬ 4: Авторизация (RBAC)                               │
│  • 4 роли: admin / chief_inspector / inspector / viewer      │
│  • Проверка прав в Middleware (Edge Runtime)                 │
│  • Дополнительная проверка прав в каждом API-обработчике     │
├──────────────────────────────────────────────────────────────┤
│  УРОВЕНЬ 5: Защита данных                                    │
│  • Bcrypt (12 rounds) для паролей                            │
│  • Параметризованные SQL-запросы через Prisma ORM            │
│  • Zod-валидация всех входных данных                         │
├──────────────────────────────────────────────────────────────┤
│  УРОВЕНЬ 6: Аудиторский след                                 │
│  • 100% охват изменений данных в таблице audit_log           │
│  • Запись IP-адреса, пользователя, старых и новых значений   │
│  • Неизменяемость на уровне СУБД                             │
└──────────────────────────────────────────────────────────────┘
```

### 6.2. Модель управления доступом (RBAC)

В системе реализована ролевая модель управления доступом (Role-Based Access Control) с четырьмя ролями.

| Роль | Код | Описание |
|------|-----|----------|
| Администратор | `admin` | Полный доступ ко всем функциям, управление пользователями, настройка системы |
| Главный ревизор | `chief_inspector` | Управление планами, аудитами, комиссиями, утверждение документов |
| Ревизор | `inspector` | Проведение аудитов, регистрация нарушений, ввод данных |
| Наблюдатель | `viewer` | Только просмотр данных и отчётов, без права редактирования |

### 6.3. Аутентификация и управление сессиями

**Процедура входа:**
1. Пользователь вводит `username` и `password` в форме `/login`.
2. Сервер извлекает запись `users` по `username`, проверяет хэш пароля через `bcryptjs.compare()`.
3. При успешной проверке сервер формирует сессионный токен: `base64url(payload) + "." + base64url(HMAC-SHA256(payload, JWT_SECRET))`.
4. Токен устанавливается в куку `auth_user` с флагами `HttpOnly; Secure; SameSite=Strict`.
5. Сессия действует 8 часов с момента последней активности.

**Верификация сессии (Middleware):**
1. На каждый запрос Edge Middleware читает куку `auth_user`.
2. Разделяет токен по последней точке на `encoded` и `signature`.
3. Вычисляет ожидаемую подпись через `crypto.subtle.verify` (Web Crypto API).
4. При несовпадении подписи возвращает HTTP 401 (API) или редирект на `/login` (страницы).

**Защита от перебора:**
- После 5 неудачных попыток входа в течение 15 минут учётная запись временно блокируется.
- Блокировка логируется в `audit_log` с IP-адресом клиента.

### 6.4. Хранение паролей

Пароли хранятся исключительно в виде bcrypt-хэша с параметром `saltRounds = 12`. Открытый текст пароля не хранится нигде в системе — ни в БД, ни в журналах, ни в переменных среды.

### 6.5. Аудиторский след

Все операции изменения данных (INSERT, UPDATE, DELETE), события аутентификации и критические административные действия записываются в таблицу `audit_log` со следующими атрибутами:

- Пользователь (`user_id`, имя пользователя).
- Тип действия (`action`).
- Затронутая таблица и запись (`table_name`, `record_id`).
- Предыдущее и новое значение (`old_value`, `new_value`) в формате JSON.
- IP-адрес клиента (`ip_address`).
- Временная метка (`created_at`) в UTC.

Таблица защищена от изменений на уровне СУБД — права `UPDATE` и `DELETE` для прикладной учётной записи не предоставляются.

### 6.6. Классификация данных

В соответствии с O'z DSt 2814:2014 система классифицирована как **КЗ-2** (класс защиты 2). Данные системы имеют гриф **ДСП** (Для служебного пользования).

| Категория данных | Класс | Меры защиты |
|-----------------|-------|-------------|
| Персональные данные военнослужащих (ПИНФЛ, паспорт) | ДСП | Шифрование транспортного уровня, RBAC, ЗРУ-547 |
| Ревизионные акты и нарушения | ДСП | Контроль доступа по ролям |
| Данные пользователей (пароли) | Конфиденциально | Bcrypt-хэш, никогда в открытом виде |
| Журнал аудита | ДСП | Неизменяемость на уровне СУБД |
| Нормативно-справочные данные | Открытые | Без дополнительных ограничений |

### 7.8. Резервное копирование

Политика резервного копирования АИС КРР предусматривает два уровня:

**Ежедневное полное резервное копирование** (запуск в 02:00):

```bash
#!/bin/bash
# /opt/scripts/backup-full.sh
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/mnt/backup/ais-krr"
DB_NAME="nazorat_taftish"
DB_USER="ais_krr_user"

pg_dump -U "$DB_USER" -Fc "$DB_NAME" \
  -f "${BACKUP_DIR}/full_${TIMESTAMP}.pgdump"

# Очистка резервных копий старше 30 дней
find "${BACKUP_DIR}" -name "*.pgdump" -mtime +30 -delete

echo "Full backup completed: ${TIMESTAMP}"
```

**Инкрементальное резервное копирование** (каждые 4 часа, 09:00–18:00 в рабочие дни):

```bash
#!/bin/bash
# /opt/scripts/backup-incremental.sh
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/mnt/backup/ais-krr/incremental"

pg_dump -U ais_krr_user -Fc nazorat_taftish \
  -f "${BACKUP_DIR}/incr_${TIMESTAMP}.pgdump"
```

**Настройка расписания (cron):**

```
# Полное резервное копирование — ежедневно в 02:00
0 2 * * * /opt/scripts/backup-full.sh >> /var/log/ais-krr-backup.log 2>&1

# Инкрементальное — каждые 4 часа в рабочее время (пн-пт)
0 9,13,17 * * 1-5 /opt/scripts/backup-incremental.sh >> /var/log/ais-krr-backup.log 2>&1
```

**Восстановление из резервной копии:**

```bash
# Полное восстановление
pg_restore -U ais_krr_user -d nazorat_taftish \
  --clean --if-exists \
  /mnt/backup/ais-krr/full_YYYYMMDD_HHMMSS.pgdump
```

Срок хранения резервных копий — **30 дней**. Резервные копии хранятся на отдельном физическом носителе (резервный сервер или NAS), изолированном от основного сервера приложения.

## 8. Руководство пользователя

### 8.1. Вход в систему

1. Открыть браузер и перейти по адресу системы (сообщается системным администратором).
2. На странице `/login` ввести логин (`username`) и пароль.
3. Нажать кнопку «Войти».
4. При успешном входе откроется главная страница (`/dashboard`).

**Важно:** После 5 неудачных попыток входа учётная запись блокируется на 15 минут. Обратитесь к системному администратору для досрочной разблокировки.

### 8.2. Руководство для роли «Администратор» (`admin`)

**Доступные функции:** все функции системы без ограничений.

**Управление пользователями** (`/admin`):
- Создание новых учётных записей: ФИО, логин, пароль, роль, подразделение.
- Редактирование профилей пользователей.
- Блокировка/разблокировка учётных записей.
- Сброс паролей.

**Просмотр журнала аудита** (`/admin` → Журнал):
- Поиск событий по пользователю, периоду, типу операции.
- Экспорт журнала в Excel.

**Управление переводами** (`/translation-management`):
- Редактирование текстов интерфейса на русском, узбекском и каракалпакском языках.
- Добавление новых ключей перевода.

**Диагностика** (`/diagnostics`):
- Проверка состояния подключения к базе данных.
- Просмотр сводки системных ресурсов.

### 8.3. Руководство для роли «Главный ревизор» (`chief_inspector`)

**Доступные функции:** планирование, управление аудитами, утверждение документов, отчётность, просмотр персонала и подразделений.

**Создание годового плана** (`/planning`):
1. Перейти в раздел «Планирование» → «Годовые планы».
2. Нажать «Создать план».
3. Заполнить: год, период охвата, орган контроля, направление контроля, тип инспекции, ответственный.
4. Сохранить план в статусе «Черновик».
5. После согласования изменить статус на «Утверждён», зафиксировать дату и утверждающего.

**Создание аудита** (`/audits`):
1. Перейти в раздел «Аудиты» → «Создать».
2. Выбрать воинскую часть, тип аудита, даты, главного ревизора.
3. Создать приказ о назначении комиссии (`/audits` → «Приказы»).
4. Сформировать состав комиссии.
5. Изменить статус аудита на «В процессе».

**Утверждение актов:**
1. Открыть завершённый аудит.
2. Просмотреть акт: замечания, нарушения, суммы.
3. Изменить статус акта на «Утверждён».

### 8.4. Руководство для роли «Ревизор» (`inspector`)

**Доступные функции:** ввод данных по аудитам, регистрация нарушений, замечаний, просмотр планов и отчётов.

**Регистрация нарушения** (`/violations`):
1. Перейти в раздел «Нарушения» → «Добавить».
2. Выбрать аудит, в рамках которого выявлено нарушение.
3. Заполнить: номер нарушения, категорию (из 14 классификаторов), описание, сумму, степень серьёзности, ответственное лицо, дату выявления.
4. Сохранить запись.

**Добавление замечания к акту:**
1. Открыть аудит → Акты → нужный акт.
2. Перейти на вкладку «Замечания».
3. Нажать «Добавить замечание», заполнить описание, степень критичности, срок устранения.

**Регистрация решения по нарушению:**
1. Открыть нарушение в реестре.
2. Нажать «Добавить решение».
3. Заполнить: номер решения, тип, описание, исполнитель, срок исполнения.

### 8.5. Руководство для роли «Наблюдатель» (`viewer`)

**Доступные функции:** только просмотр и экспорт данных.

- Просмотр планов ревизий, аудитов, нарушений и решений в режиме «только чтение».
- Генерация и скачивание отчётов в форматах Excel и PDF.
- Просмотр аналитических диаграмм на дашборде.
- Поиск по реестрам нормативных документов.

> **Наблюдатель** не имеет права создавать, редактировать или удалять записи. Кнопки «Создать», «Редактировать», «Удалить» для данной роли не отображаются.

### 9.4. Мониторинг

Рекомендуется настроить следующий мониторинг:

- **PM2 Monitoring:** `pm2 monit` — процессор, память, количество перезапусков.
- **PostgreSQL:** мониторинг активных соединений (`pg_stat_activity`), размера таблиц, медленных запросов.
- **Nginx:** анализ access.log и error.log на аномальный трафик.
- **Сервер:** утилизация CPU, RAM, дискового пространства (рекомендуется Prometheus + Grafana или аналог).
- **Журнал аудита:** ежедневный просмотр системным администратором.

### Б.2. Специальные ограничения

| Ограничение | Описание |

### Б.3. Реализация RBAC

Проверка прав реализована на двух уровнях:

1. **Edge Middleware (`middleware.ts`):** перехватывает все запросы, верифицирует HMAC-подпись сессии. Неаутентифицированные запросы отклоняются до обращения к серверному коду.

2. **API-обработчики (`/app/api/**`):** каждый обработчик извлекает роль пользователя из сессии и проверяет наличие необходимых прав. При нехватке прав возвращается HTTP 403 с сообщением об ошибке.

Роль пользователя хранится в поле `users.role` базы данных и включается в сессионный токен при аутентификации. Изменение роли вступает в силу при следующем входе в систему.
