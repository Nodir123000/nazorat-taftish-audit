## Ж.1. НАЗНАЧЕНИЕ И ОБЛАСТЬ ПРИМЕНЕНИЯ

Настоящее приложение определяет матрицу разграничения доступа (RBAC) для ролей пользователей АИС «КРР». Матрица обеспечивает реализацию принципа минимальных привилегий: каждый пользователь имеет доступ только к тем функциям и данным, которые необходимы для выполнения его должностных обязанностей.

**Принципиальное разделение ролей:**

1. **Операционные роли** (Инспектор, Старший инспектор, Руководитель инспекции, Руководитель КРУ) — имеют полный набор прав: просмотр (V), создание (C), редактирование (E), удаление (D), утверждение (A) в рамках своей зоны ответственности.

2. **Аналитические роли** (Финансовая служба В/Ч, Финансовая служба Военного округа, ГФЭУ МО РУз, Службы ВО и управления МО РУз) — имеют **исключительно** право просмотра (V) и экспорта отчётов (E) в пределах своей зоны ответственности. Права создания (C), редактирования (E) и удаления (D) ревизий и планов для аналитических ролей **категорически запрещены**. Это требование реализовано на уровне RBAC-политик и RLS в СУБД PostgreSQL, что исключает любую возможность создания или модификации ревизий аналитическими ролями — в том числе через прямые SQL-запросы.

## Ж.4. ОБОЗНАЧЕНИЯ И ПРАВИЛА

### Ж.4.1. Обозначения прав доступа

| Обозначение | Наименование | Описание |
|-------------|-------------|---------|
| **V** | View (Просмотр) | Доступ к чтению данных и отчётов |
| **C** | Create (Создание) | Создание новых записей (ревизий, нарушений, планов) |
| **E** | Edit (Редактирование) | Изменение существующих записей |
| **D** | Delete (Удаление) | Физическое удаление записей |
| **A** | Approve (Утверждение) | Утверждение документов, планов, актов |
| **—** | Нет доступа | Функция недоступна для данной роли |
| **V (Audit)** | Просмотр аудита | Доступ только к журналу аудита (без возможности редактирования) |

### Ж.4.2. Правила расшифровки

1. **Для ПС-3 (Ревизии):** Инспектор создаёт ревизии только по своей учетной записи. Старший инспектор — по всем инспекторам своей группы.
2. **Для ПС-4 (Нарушения):** Право на создание и редактирование нарушений принадлежит инспекторам и руководителям КРУ. Аналитические роли (Фин. службы, ГФЭУ, Службы ВО) — только просмотр.
3. **Для ПС-6 (Отчётность):** Все аналитические роли имеют право на экспорт (E) отчётов в XLSX/PDF в пределах своей зоны ответственности.
4. **Для ПС-8 (Администрирование/ИБ):** Ни одна из аналитических ролей не имеет права создания, редактирования или удаления пользователей и настроек. Доступ — только к журналу аудита.
5. **Для всех аналитических ролей:** Функции C, E, D, A — **полностью запрещены** для всех подсистем без исключения.

## Ж.5. ФИЛЬТРАЦИЯ ДАННЫХ ПО РОЛЯМ (RLS)

### Таблица Ж.5.1 — Зоны видимости данных

| Роль | Видимость данных | Ограничение на уровне строк (RLS) |
|------|-----------------|----------------------------------|
| Инспектор | Свои ревизии, нарушения, акты | `unit_id = current_user.unit_id AND user_id = current_user.id` |
| Старший инспектор | Ревизии группы | `unit_id = current_user.unit_id AND group_id = current_user.group_id` |
| Руководитель инспекции | Все данные инспекции | `unit_id = current_user.unit_id` |
| Руководитель КРУ | Вся республика | Без ограничений (все данные) |
| Администратор ИС | Технические данные, логи | Без ограничений (системные данные) |
| **Финансовая служба В/Ч** | Данные только по своей В/Ч | `unit_id = current_user.unit_id` |
| **Финансовая служба ВО** | Данные по всем В/Ч своего округа | `district_id = current_user.district_id` |
| **ГФЭУ МО РУз** | Все данные республики | Без ограничений (полный доступ к аналитике) |
| **Службы ВО и управления МО РУз** | Данные по своему направлению | `service_branch = current_user.service_branch` |

### Ж.5.2. Реализация RLS

Фильтрация на уровне строк (Row-Level Security) реализуется на уровне СУБД PostgreSQL 15 через политики `pg_policy`. Каждый запрос автоматически дополняется предикатом, ограничивающим выборку в соответствии с зоной ответственности пользователя. Попытка обхода RLS через прямые SQL-запросы технически исключена архитектурой СУБД.

## Ж.6. РАЗДЕЛЕНИЕ ПРАВ МЕЖДУ РУКОВОДИТЕЛЯМИ И АНАЛИТИКАМИ

### Ж.6.1. Ключевое различие

| Операция | Руководитель КРУ | ГФЭУ МО РУз |
|----------|-----------------|-------------|
| Создание ревизии | **Да** | **Нет** |
| Редактирование ревизии | **Да** | **Нет** |
| Утверждение акта | **Да** | **Нет** |
| Просмотр всех данных | **Да** | **Да** |
| Формирование сводных отчётов | **Да** | **Да** |
| Экспорт данных | **Да** | **Да** |
| Доступ к Audit Log | **Да** | **Да** (только чтение) |

### Ж.6.2. Обоснование разделения

Аналитические роли (Финансовые службы В/Ч и округов, ГФЭУ, Службы ВО и управления МО РУз) предназначены исключительно для мониторинга и анализа. Наделение их правом создания или редактирования ревизий создаёт конфликт интересов и нарушает принцип разделения функций контроля и финансового управления.

Право создания и редактирования ревизий принадлежит исключительно:
- Инспекторам и старшим инспекторам КРУ;
- Руководителям инспекций;
- Руководителю КРУ.

**Техническая реализация ограничений для аналитических ролей:** запрет на создание (C), редактирование (E) и удаление (D) для аналитических ролей реализован на двух уровнях:
1. **RBAC (уровень приложения):** маршруты API `/api/audits/*`, `/api/planning/*` и связанные маршруты для аналитических ролей доступны только с HTTP-методами GET и OPTIONS. Методы POST, PUT, PATCH, DELETE возвращают `403 Forbidden`. Реализация — функция `withAuth(handler, subsystem, action)` (`lib/security/rbac-guardian.ts`) с проверкой на основании матрицы Ж.3.1.
2. **RLS (уровень СУБД):** политики PostgreSQL `pg_policy` на таблицах `audits`, `rev_plan_year`, `violations` и других операционных таблицах содержат условие `current_user_role NOT IN ('financial_unit', 'financial_district', 'gfeu', 'service_mo')` для операций C/E/D. Даже при попытке прямого SQL-доступа (например, через скомпрометированный аккаунт администратора БД) политики RLS блокируют операции модификации.

1. На этапе 3.5 проводится автоматизированный тест: для каждой пары (роль × подсистема × HTTP-метод) проверяется ожидаемый код ответа (`200 OK` / `403 Forbidden`).

3. Покрытие — обязательная часть приёмочных испытаний (Приложение Б, ПМИ).
