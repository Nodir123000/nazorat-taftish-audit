## Ё.2. МОДЕЛЬ УГРОЗ STRIDE

### Ё.2.1. Классификация угроз по STRIDE

| Категория | Описание | Примеры угроз | Уровень риска |
|-----------|---------|--------------|--------------|
| **S** — Spoofing | Подмена идентичности | Кража учётных данных, фишинг | Высокий |
| **T** — Tampering | Изменение данных | SQL-инъекция, подделка документов | Критический |
| **R** — Repudiation | Отказ от действий | Удаление записей без логирования | Средний |
| **I** — Information Disclosure | Раскрытие информации | Утечка персональных данных | Критический |
| **D** — Denial of Service | Отказ в обслуживании | DDoS, перегрузка системы | Средний |
| **E** — Elevation of Privilege | Повышение привилегий | Взлом учётки, privilege escalation | Критический |

### Ё.5.3. Соответствие O'zMSt 270-2024

Настоящий раздел устанавливает соответствие криптографических мер защиты, реализованных в АИС «КРР», требованиям национального стандарта O'zMSt 270-2024 «Информационная технология. Информационная безопасность. Данные для криптографической защиты. Алгоритмы шифрования».

#### Таблица Ё.5.3.1 — Соответствие алгоритмов

| Требование O'zMSt 270-2024 | Реализация в АИС «КРР» | Статус |
|--------------------------|----------------------|--------|
| **Алгоритм 1 (MShA):** 256-битный блок, ключ 256/512 бит, режим CBC | Не применяется (выбран AES) | ⚠️ Принято |
| **Алгоритм 2 (AES/Rijndael):** 128-битный блок, ключ 128/192/256 бит | AES-256-GCM / AES-256-CBC | ✅ Соответствует |
| **Алгоритм 3:** Кriptографическое отображение, гамширование | Не применяется (GCM обеспечивает аналогичную защиту) | ⚠️ Принято |
| **Длина ключа ≥ 256 бит** (для симметричных алгоритмов) | AES-256 (256 бит) | ✅ Соответствует |
| **Режим CBC** (для MShA) | AES-256-GCM (режим с аутентификацией) | ✅ Допустимо (GCM ≥ CBC по безопасности) |

## Ё.8. РЕГЛАМЕНТ ПРИЁМА ДАННЫХ С ВНЕШНИХ НОСИТЕЛЕЙ

Ввиду отсутствия внешних сетевых интеграций (режим Air-gapped), первичный внос данных в АИС «КРР» осуществляется исключительно через сертифицированные USB-носители.

### Ё.8.1. Требования к процедуре импорта

1. **Автоматическая проверка:** Любой файл (XLSX, PDF, CSV), импортируемый в систему, должен проходить автоматическое антивирусное сканирование встроенными средствами защиты.
2. **Изолированная среда («песочница»):** Обработка и парсинг файлов (извлечение данных из таблиц Excel, метаданных PDF) должны выполняться в изолированном программном контейнере (Sandbox), не имеющем прямого доступа к основной базе данных.
3. **Валидация перед записью:** Запись данных в основную БД допускается только после успешного прохождения проверки структуры файла и отсутствия вредоносного кода.
4. **Регистрация вноса:** Каждый факт импорта данных с внешнего носителя должен фиксироваться в журнале аудита с указанием ФИО ответственного, ID носителя и результата проверки на вирусы.
5. **Шифрование импортируемых данных:** Данные, импортируемые с внешних носителей, подлежат шифрованию в соответствии с п. 4.1.6.4 ТЗ и разделом Ё.7 настоящего приложения перед записью в БД.

### Ё.8.2. Дополнительные меры защиты при импорте

| Мера | Описание | Реализация | Приоритет |
|------|---------|------------|----------|
| **Проверка носителя** | Предварительное сканирование USB-носителя на вирусы | Антивирусное ПО с обновлёнными базами | P1 |
| **Логирование подключений** | Фиксация подключения USB-устройств в Audit Log | Audit Log (событие `USB_DEVICE_CONNECTED`) | P1 |
| **Ограничение типов файлов** | Допускаются только XLSX, PDF, CSV | Фильтрация по MIME-типу | P1 |
| **Контрольhash-записи** | Проверкаhash-суммы импортируемого файла до и после копирования | SHA-256 файла сравнивается до и после | P2 |
| **Уничтожение данных с носителя** | После импорта данные с USB-носителя подлежат безопасному удалению | NIST SP 800-88 | P2 |
