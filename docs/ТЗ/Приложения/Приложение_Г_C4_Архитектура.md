## Г.1. ОБЩИЕ ПОЛОЖЕНИЯ

### Г.1.1. Назначение

Настоящее приложение описывает архитектуру АИС КРУ МО РУЗ с использованием модели C4 (Context, Container, Component, Code) для визуализации структуры системы на различных уровнях детализации.

### Г.1.2. Уровни абстракции C4

| Уровень | Назначение | Аудитория |
|---------|-----------|-----------|
| **Context (C4-1)** | Общий обзор системы и её окружения | Менеджеры, заинтересованные стороны |
| **Container (C4-2)** | Контейнеры (приложения, сервисы) | Разработчики, DevOps |
| **Component (C4-3)** | Компоненты внутри контейнеров | Разработчики |
| **Code (C4-4)** | Детали реализации (опционально) | Разработчики |

## Г.3. УРОВЕНЬ КОНТЕЙНЕРОВ (C4-2)

### Г.3.1. Архитектура контейнеров

#### Таблица контейнеров системы (базовая поставка)

| № | Контейнер | Технология | Назначение | Развёртывание |
|----|-----------|------------|-----------|--------------|
| 1 | **Web + API (Next.js full-stack)** | Next.js 16, React 19, TypeScript 5 | Пользовательский интерфейс (SSR), API Routes, бизнес-логика, расчёт KPI (MathJS) | Container |
| 2 | **Reverse Proxy** | nginx | Завершение TLS 1.3, балансировка, маршрутизация | Container |
| 3 | **Database** | PostgreSQL 15 (`pgcrypto`, RLS) | Основное хранилище данных | VM / Container |
| 4 | **ORM Layer** | Prisma 7 | Доступ к БД, миграции | Внутри Web+API |
| 5 | **Cache & Sessions** | Redis 7 (`ioredis`) | Кэширование, сессии, очередь фоновых задач | Container |
| 6 | **File Storage** | MinIO / S3-совместимое (`@aws-sdk/client-s3`) | Хранение сканов и приложений | Container |

#### Опциональные контейнеры (Стадия 3, Итерация 4 — по согласованию с Заказчиком)

| № | Контейнер | Технология | Назначение | Условие включения |
|----|-----------|------------|-----------|--------------------|
| 7 | **Key Management** | HashiCorp Vault или локальный KMS на pgcrypto | Хранение и ротация ключей AES-256-GCM | При классе защиты КЗ-2 |
| 8 | **Search Engine** | PostgreSQL FTS (`tsvector`) или Elasticsearch | Полнотекстовый поиск по реестрам | По объёму данных |
| 9 | **Container Orchestrator** | Kubernetes / Docker Swarm | Горизонтальное масштабирование | При росте нагрузки > 500 пользователей |

## Г.5. ДИАГРАММЫ ПОСЛЕДОВАТЕЛЬНОСТИ (SEQUENCE)

> Приведённые ниже сценарии соответствуют сценариям использования UC-01…UC-03, определённым в п. 4.1.1.3 основного ТЗ. Авторизация пользователей и генерация отчётов рассматриваются как вспомогательные сценарии и приведены в Г.5.4–Г.5.5.

### Г.5.1. Sequence: UC-01 «Регистрация ревизии»

| Шаг | Участник | Действие | Результат |
|-----|---------|---------|----------|
| 1 | Инспектор | Нажатие «Создать ревизию» в ПС-3 | Открытие формы |
| 2 | Web App | Заполнение реквизитов формы (объект, период, основание) | unit_id, plan_id |
| 3 | Web App | POST /api/audits | Запрос создания |
| 4 | AuditController (Next.js API Route) | Валидация Zod-схемы | Проверка полей |
| 5 | ConflictChecker | Проверка конфликта интересов по ШДК | Нет конфликта |
| 6 | PlanRepository (Prisma) | Проверка существования плана | План `rev_plan_year` существует |
| 7 | AuditRepository (Prisma) | INSERT в `audits` с генерацией номера | audit_number, audit_id |
| 8 | CommissionManager | INSERT в `commission_members` | Состав комиссии создан |
| 9 | AuditLogger | INSERT в `audit_log` (с SHA-256 chaining) | Событие AUDIT_CREATED |
| 10 | API | Возврат ответа | 201 Created |
| 11 | Web App | Отображение сообщения | «Ревизия создана» |

### Г.5.2. Sequence: UC-02 «Расчёт KPI инспектора»

| Шаг | Участник | Действие | Результат |
|-----|---------|---------|----------|
| 1 | Планировщик / Руководитель КРУ | Запуск задачи расчёта KPI (по cron или вручную) | Job создана |
| 2 | KPIController (Next.js API Route) | Чтение данных из ПС-1, ПС-3, ПС-4 за период | Массив показателей |
| 3 | KPIController | Получение нормативных весов из ПС-7 | weights[6 категорий] |
| 4 | KPIEngine (MathJS 15) | Расчёт `KPI = Σ(Wi × Ni) / ΣWi` для каждого инспектора | Значения KPI ≥ 0.01 |
| 5 | KPIRepository (Prisma) | INSERT/UPDATE в `user_kpi_scores` | Результаты сохранены |
| 6 | NotificationService | При KPI < 75% — генерация уведомления `KPI_CRITICAL_THRESHOLD` | Запись в `notifications` |
| 7 | AuditLogger | INSERT в `audit_log` | Событие KPI_CALCULATED |
| 8 | Dashboard API | Формирование агрегатов для дашборда | Кэш в Redis |
| 9 | Руководитель КРУ | Просмотр рейтинговой таблицы и графиков | Web UI отображает результаты |

### Г.5.3. Sequence: UC-03 «Управление локализацией интерфейса»

| Шаг | Участник | Действие | Результат |
|-----|---------|---------|----------|
| 1 | Пользователь | Выбор языка (RU / UZ-кирилл / UZ-лат) в настройках профиля | locale |
| 2 | Web App | PUT /api/users/me/locale | Сохранение выбора |
| 3 | UserRepository (Prisma) | UPDATE `users.locale` | Профиль обновлён |
| 4 | i18n Service | Загрузка пакета переводов из `i18n_keys` + `i18n_values` | Hot reload без перезагрузки |
| 5 | Web App | Перерисовка интерфейса с новой локалью | Время ≤ 0.5 с |
| 6 | AuditLogger | INSERT в `audit_log` | Событие LOCALE_CHANGED |

### Г.5.4. Вспомогательный сценарий: Авторизация пользователя

## Г.6. ER-ДИАГРАММА (УПРОЩЁННАЯ)

### Г.6.1. Основные сущности и связи

| № | Сущность | Описание | Ключевые атрибуты | Связи |
|---|----------|---------|-------------------|-------|
| 1 | **users** | Пользователи системы | id, login, email, role, unit_id | 1:N → audits, 1:N → audit_log |
| 2 | **units** | Подразделения (воинские части) | id, name, code, region_id | 1:N → audits, 1:N → personnel |
| 3 | **annual_plans** | Годовые планы | id, year, status | 1:N → quarterly_plans |
| 4 | **quarterly_plans** | Квартальные планы | id, annual_plan_id, quarter | 1:N → monthly_plans |
| 5 | **monthly_plans** | Месячные планы | id, quarterly_plan_id, month | 1:N → audits |
| 6 | **audits** | Ревизии | id, monthly_plan_id, unit_id, status | 1:N → audit_reports, 1:N → violations |
| 7 | **audit_reports** | Акты проверок | id, audit_id, report_number | 1:N → findings |
| 8 | **violations** | Нарушения | id, audit_id, type, amount | 1:N → decisions |
| 9 | **decisions** | Решения по нарушениям | id, violation_id, status | |
| 10 | **orders** | Приказы | id, order_number, plan_id | |
| 11 | **commission_members** | Члены комиссий | id, audit_id, user_id, role | |
| 12 | **audit_log** | Журнал аудита | id, user_id, action, table_name | |

### Г.6.2. Таблица связей между сущностями

| № | Сущность A | Связь | Сущность B | Тип связи |
|---|-----------|-------|-----------|----------|
| 1 | annual_plans | включает | quarterly_plans | 1:N |
| 2 | quarterly_plans | включает | monthly_plans | 1:N |
| 3 | monthly_plans | содержит | audits | 1:N |
| 4 | units | содержит | audits | 1:N |
| 5 | users | создаёт | audits | 1:N |
| 6 | audits | имеет | audit_reports | 1:N |
| 7 | audit_reports | содержит | findings | 1:N |
| 8 | audits | фиксирует | violations | 1:N |
| 9 | violations | имеют | decisions | 1:N |
| 10 | orders | назначает | audits | 1:N |
| 11 | users | входят в | commission_members | 1:N |
| 12 | audits | включают | commission_members | 1:N |
| 13 | units | содержат | personnel | 1:N |
| 14 | personnel | связаны с | users | 1:1 |

                    ┌─────────────────────────┼─────────────────────────┐
                    │                         │                         │
