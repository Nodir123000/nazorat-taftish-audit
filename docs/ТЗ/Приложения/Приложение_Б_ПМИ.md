## Б.1. ОБЩИЕ ПОЛОЖЕНИЯ

### Б.1.1. Назначение и область применения

Настоящая программа и методика испытаний (ПМИ) устанавливает порядок и методы проведения испытаний АИС «КРР» для подтверждения соответствия системы требованиям технического задания (ТЗ) и требованиям ГОСТ 34.603-92 «Информационная технология. Виды испытаний автоматизированных систем».

ПМИ применяется на всех этапах испытаний, определённых в Разделе 6 ТЗ.

### Б.1.2. Нормативные ссылки

- ГОСТ 34.603-92 — Виды испытаний автоматизированных систем
- ГОСТ 34.912.1-2013 — Оценка качества автоматизированных систем
- O'z DSt 1987:2018 — Техническое задание на создание информационной системы
- O'z DSt 2814:2014 — Классификация по уровню защищённости от НСД

### Б.1.3. Виды и этапы испытаний

Испытания АИС «КРР» проводятся в четыре этапа в соответствии с Разделом 6 ТЗ:

| Этап | Наименование | Назначение | Метод |
|------|--------------|------------|-------|
| **Этап 1** | Автономные испытания | Предварительная проверка каждой подсистемы и инфраструктурных компонентов отдельно | Функциональное и нагрузочное тестирование |
| **Этап 2** | Комплексные испытания | Предварительная проверка взаимодействия всех подсистем и end-to-end сценариев | Интеграционное тестирование |
| **Этап 3** | Опытная эксплуатация | Работа в реальных условиях на ограниченном наборе объектов (45 дней) | Мониторинг в реальной среде |
| **Этап 4** | Приёмочные испытания | Финальная проверка на соответствие ТЗ комиссией МО РУз | Комиссионные испытания по ПМИ |

> **Примечание.** В соответствии с ГОСТ 34.603-92 и О'z DSt 1987:2018 опытная эксплуатация проводится как самостоятельный вид испытаний после предварительных (автономных и комплексных) и предшествует приёмочным испытаниям. Сроки и порядок опытной эксплуатации определены в Разделе 6 ТЗ и п. 4.1 стадии 4 Календарного плана (Приложение А).

### Б.1.4. Объекты испытаний

| № | Объект | Тип | Подсистема / Компонент |
|---|--------|-----|------------------------|
| 1 | Модуль расчёта KPI | Автономный | **ПС-2** — Мониторинг эффективности (KPI) |
| 2 | Модуль учёта ревизий (MinIO) | Автономный | **ПС-3** — Ревизии |
| 3 | Модуль локализации | Автономный | **ПС-9** — Локализация |
| 4 | Elasticsearch (поиск) | Автономный | **ПС-7** — НСИ |
| 5 | HashiCorp Vault (ключи) | Автономный | **ПС-8** — Администрирование/ИБ |
| 6 | RabbitMQ (очереди) | Комплексный | **ПС-2**, **ПС-6** |
| 7 | PostgreSQL (failover) | Восстановление | Инфраструктура |
| 8 | MinIO (отказ хранилища) | Восстановление | Инфраструктура |

### Б.1.5. Критерии завершения испытаний

Испытания считаются завершёнными при выполнении следующих условий:
- 100% сценариев автономных испытаний пройдены успешно;
- 100% комплексных сценариев пройдены успешно;
- 100% сценариев восстановления пройдены успешно (RTO и RPO соответствуют требованиям п. 4.1.5 ТЗ);
- протоколы испытаний подписаны комиссией.

## Б.2. АВТОНОМНЫЕ ИСПЫТАНИЯ

### Сценарий №1. Проверка расчёта KPI (ПС-2)

**Цель:** Верификация безопасности и точности расчёта баллов через MathJS (без eval).

**Объект:** ПС-2 KPI Engine (Модуль «Алгоритмы MathJS»), MathJS Engine, Analytics API (FastAPI).

**Действие:**
1. Инспектор вводит показатели за месяц: объём нарушений (500), количество актов (12), доля закрытых предписаний (0.85).
2. Система применяет формулу `KPI = Σ(Wi × Ni) / ΣWi` из БД.
3. Попытка внедрения вредоносного JS-кода: в поле формулы вводится `eval("alert('hack')")`.

**Ожидаемый результат:**
- Система выводит корректный результат: KPI рассчитан с точностью до 0.01 балла.
- Внедрение вредоносного JS-кода блокируется парсером формул (MathJS Tokenizer); сообщение об ошибке отображается пользователю.
- Время расчёта для 500 инспекторов — не более 60 с.
- Результат заносится в `user_kpi_scores`, журналируется в Audit Log (SHA-256 chaining).

**Критерий успеха:** ✅ Расчёт выполнен корректно; SQL/XSS-инъекция заблокирована.

### Сценарий №2. Регистрация внеплановой проверки (ПС-3)

**Цель:** Проверка безопасного доступа к файлам в MinIO; верификация механизма предварительно подписанных URL (Pre-signed).

3. Пользователь нажимает «Скачать» — пытается получить файл по прямой ссылке без авторизации.

**Ожидаемый результат:**
- Файл успешно загружен; запись в `unplanned_audits.document_s3_key` создана.
- Pre-signed URL действует ровно 15 минут; по истечении — доступ запрещён.
- Прямой доступ к S3-бакету (`minio://ais-krr-files/`) запрещён через политику bucket policy.
- Срок жизни ссылки указан в логе: `presigned_url_expires_at = NOW() + 15min`.

**Критерий успеха:** ✅ Ссылка временная; прямой доступ заблокирован.

### Сценарий №5. Управление ключами через Vault KMS (ПС-8)

**Цель:** Проверка безопасного хранения секретов (AES-256) и выдачи ключей авторизованным сервисам.

**Объект:** HashiCorp Vault 1.x, ПС-8 Администрирование/ИБ (Модуль «Vault KMS»).

**Действие:**
1. Сервис генерации отчётов запрашивает ключ AES-256 для расшифровки документа.
2. Vault проверяет политику доступа сервиса (AWS IAM / Kubernetes ServiceAccount).
3. Неавторизованный контейнер (без JWT) пытается получить ключ.

**Ожидаемый результат:**
- Авторизованный сервис получает ключ AES-256 через Vault API; ключ не хранится в логах.
- Попытка неавторизованного доступа отклонена; событие записано в audit log Vault.
- Срок действия токена — 1 час; по истечении — требуется ре-аутентификация.
- Ключи ротируются автоматически (политика Vault: rotate каждые 90 дней).

**Критерий успеха:** ✅ Ключ выдан авторизованному сервису; неавторизованный доступ заблокирован.

## Б.3. КОМПЛЕКСНЫЕ ИСПЫТАНИЯ

### Сценарий №6. Интеграционное тестирование RabbitMQ (ПС-2, ПС-6)

**Цель:** Проверка асинхронной обработки тяжёлых задач (генерация отчётов, расчёт KPI) через RabbitMQ; подтверждение отсутствия блокировки UI.

**Объект:** RabbitMQ 3.x, ПС-2 Мониторинг эффективности (KPI), ПС-6 Отчётность (Модуль «Экспорт данных»), nginx.

**Действие:**
1. Руководитель КРУ запрашивает генерацию сводного отчёта за квартал (расчёт KPI по 500 инспекторам, 10 000 нарушений).
2. Запрос публикуется в очередь `reports.generate` через nginx API Gateway.
3. Пользователь выполняет навигацию по интерфейсу (открывает другую страницу) — UI не заблокирован.
4. Фоновая задача обрабатывается Consumer-сервисом; результат сохраняется в MinIO; пользователю отправляется уведомление.

**Ожидаемый результат:**
- UI не заблокирован; страница откликается во время генерации отчёта (отклик < 3 с).
- Задача обработана асинхронно; очередь `reports.generate` содержит сообщение со статусом `pending → processing → completed`.
- После завершения — пользователь получает уведомление (badge или push); файл доступен для скачивания.
- При остановке RabbitMQ на 5 минут — накопленные сообщения обработаны после восстановления; no message loss.

**Критерий успеха:** ✅ UI не заблокирован; асинхронная обработка работает; no message loss.

## Б.4. ИСПЫТАНИЯ НА ВОССТАНОВЛЕНИЕ (ОТКАЗОУСТОЙЧИВОСТЬ)

### Сценарий №7. Отказ Primary-узла PostgreSQL (Failover)

**Цель:** Проверка автоматического восстановления после сбоя primary-узла СУБД; подтверждение RTO ≤ 1 час, RPO ≤ 15 мин.

**Объект:** PostgreSQL 15 (Master-Slave replication), ИБП.

**Действие:**
1. Имитируется отказ primary-узла: отключение питания, остановка сервиса (`pg_ctl stop -m fast`) на 300 секунд.
2. Система фиксирует время потери связи с primary; автоматический триггер promoted standby.
3. После восстановления primary — проверяется целостность данных (WAL replay).

**Ожидаемый результат:**
- Failover на реплику выполнен не более чем за 60 секунд.
- RPO ≤ 15 минут — все транзакции, подтверждённые до момента отказа, сохранены.
- Приложение автоматически переподключается к новому primary через PgBouncer.
- Корректное завершение текущих транзакций — ни одна транзакция не потеряна.
- После восстановления original primary — репликация возобновлена, отставание ликвидировано.

**Критерий успеха:** ✅ Failover < 60 сек; RPO < 15 мин; данные не потеряны.

### Сценарий №10. Отказ питания (ИБП) — комплексный

**Цель:** Проверка корректного завершения транзакций и сохранности данных при сбое электропитания.

**Объекты:** ИБП (30 мин автономность), PostgreSQL, Redis, Docker.

**Действие:**
1. Имитируется сбой электропитания — ИБП переключается на автономное питание.
2. Система получает сигнал `onBattery` от UPS через SNMP.
3. Через 25 минут ИБП сигнализирует о критическом уровне заряда; выполняется корректное завершение (graceful shutdown) всех сервисов.

**Ожидаемый результат:**
- Все транзакции PostgreSQL завершены (ACID); WAL записан.
- Redis данные сессий сохранены (`BGREWRITEAOF` выполнен).
- Docker контейнеры остановлены корректно (`docker stop -t 60`).
- После восстановления питания — все сервисы запущены автоматически; данные не потеряны.
- Журнал событий (`audit_log.type = 'POWER_FAILURE'`) содержит запись о сбое.

**Критерий успеха:** ✅ Корректное завершение; данные сохранены; автоматический restart.

### Сценарий №12. Интеграция с ШДК МО РУз (Единый источник данных)

**Цель:** Проверка корректности импорта данных о персонале и оргструктуре из внутренней «Штатно-должностной книги» (ШДК) в рамках изолированного контура.

**Действие:**

2. Проверяется соответствие импортированных данных (ФИО, PINFL, Должность) исходным данным в ШДК.
3. Имитируется попытка импорта некорректных данных (отсутствие обязательного поля ПИНФЛ).

**Ожидаемый результат:**
- Данные импортированы без ошибок; в ПС-5 созданы/обновлены цифровые досье.

- Время выполнения полного импорта справочника (1000+ записей) — не более 5 минут.
- Все операции импорта отражены в Audit Log.

**Критерий успеха:** ✅ Данные в АИС КРР идентичны данным ШДК; целостность сохранена.

### Сценарий №13. Автоматическая блокировка плана (Иммутабельность)

2. Система выполняет запрос к ШДК — данных о конфликте не найдено.

### Сценарий №18. Проверка SHA-256 верификации и срабатывание уведомления о нарушении целостности

**Объекты:** ПС-10.3 (Мои ревизии), ПС-8 (Audit Log), ПС-10 (Центр уведомлений).

2. Зафиксирована запись нарушения с violation_id=`V-2026-00042`, созданная через карточку инспектора. Хэши: `violations.content_hash = H1`, `audits.violation_hash_ref = H2`.

**Действие (Имитация изменения напрямую в БД):**
1. Администратор (для целей тестирования) напрямую модифицирует запись в таблице `violations`: изменяет поле `amount` с `200000.00` на `250000.00` SQL-командой:
   ```sql
   UPDATE violations SET amount = 250000.00 WHERE id = 'V-2026-00042';
   ```
2. Запускается фоновый процесс Check Integrity (или ожидание до следующего цикла, 5 минут).
3. Процесс сверяет хэши: расчётное значение `SHA256(...)` ≠ сохранённому в `audits.violation_hash_ref`.

**Ожидаемый результат (Срабатывание верификации):**
- Check Integrity обнаруживает несоответствие: `violations.content_hash` не совпадает с расчётным значением.
- Генерируется событие `DATA_INTEGRITY_VIOLATION` в Audit Log (SHA-256 chaining).
- Создаётся запись в `security_incidents`: тип=`INTEGRITY_MISMATCH`, severity=`CRITICAL`, violation_id=`V-2026-00042`.
- Генерируется уведомление администратору ИБ (ПС-10): «Обнаружено расхождение целостности данных. Нарушение ID V-2026-00042. Возможна несанкционированная модификация.»
- Статус ревизии в реестре `audits/financial` помечается как `'integrity_alert'`.

**Ожидаемый результат (Карточка инспектора):**
- При открытии карточки инспектора Ж. в разделе «Мои ревизии» отображается предупреждение: «Внимание: обнаружено несоответствие данных. Обратитесь к администратору.»

**Критерий успеха:** ✅ SHA-256 mismatch обнаружен; событие DATA_INTEGRITY_VIOLATION зафиксировано; инцидент ИБ создан; уведомление доставлено администратору.
