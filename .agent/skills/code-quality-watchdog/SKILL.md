# Code Quality Watchdog — Инструкции для Агента

## Когда использовать

Запускай аудит **ПЕРЕД** тем, как объявить задачу выполненной:
- После создания нового компонента или страницы
- После реализации нового API-эндпоинта
- После изменения схемы Prisma
- По запросу пользователя (`npm run audit`)

## Как запустить

```bash
npx tsx .agent/skills/code-quality-watchdog/scripts/quality-audit.ts
```

Или через npm-скрипт:
```bash
npm run audit
```

## Что проверяет скрипт

### ✅ FRONTEND (TSX/TSX-компоненты)
| Правило | Критичность | Описание |
|---------|------------|----------|
| NO_DIRECT_FETCH | CRITICAL | Запрет `fetch()` в компонентах напрямую — использовать хуки или SWR |
| NO_HARDCODE_TEXT | WARNING | Кириллические строки должны быть в i18n или константах |
| USE_CLIENT_MARK | WARNING | `'use client'` обязателен в интерактивных компонентах |
| NO_INLINE_PRISMA | CRITICAL | Prisma никогда не используется на клиенте |

### ✅ BACKEND (API Routes)
| Правило | Критичность | Описание |
|---------|------------|----------|
| USE_SHARED_PRISMA | CRITICAL | Использовать только `import prisma from '@/lib/db/prisma'` |
| NO_PRISMA_NEW | CRITICAL | Запрет `new PrismaClient()` — использовать только singleton |
| SERVICE_LAYER | WARNING | Бизнес-логику выносить в `lib/services/`, а не держать в route.ts |
| HANDLE_ERRORS | WARNING | Все API должны иметь try/catch с возвратом статусов |

### ✅ DATABASE (Prisma Schema)
| Правило | Критичность | Описание |
|---------|------------|----------|
| HAS_CREATED_AT | WARNING | Все модели должны иметь поле `created_at` |
| HAS_STATUS | INFO | Рекомендуется поле `status` для управляемых сущностей |
| DECIMAL_CAST | WARNING | Поля Decimal при возврате в JSON конвертировать через `Number()` |

## Интерпретация результатов

- **CRITICAL** — код НЕ должен попасть в продакшн без исправления. Исправь немедленно.
- **WARNING** — архитектурное нарушение. Исправить перед завершением задачи.
- **INFO** — рекомендация для будущих улучшений.

## Автоматическое исправление

После получения отчета, если есть CRITICAL нарушения:
1. Прочитай файл нарушения через `view_file`
2. Примени исправление через `replace_file_content`
3. Запусти аудит повторно для подтверждения
