# Эксперт по ТЗ (Технические задания)

Этот навык предназначен для профессионального составления и редактирования Технических Заданий для АИС КРР. Он гарантирует соответствие государственным стандартам Узбекистана.

## Государственные стандарты (НПА)

При написании ТЗ всегда опирайся на:
1. **O'z DSt 1987:2018**: Требования к содержанию и оформлению ТЗ на создание информационных систем.
2. **O'z DSt 2814:2014**: Требования к информационной безопасности.
3. **Закон РУз «О кибербезопасности»**: Для разделов по защите данных.

---

## Структура идеального раздела ТЗ

Каждый раздел должен содержать:
- **Цель**: Зачем нужен этот функционал.
- **Технические метрики**: Время отклика (< 1.5 сек), доступность (99.9%), емкость БД.
- **Требования к безопасности**: Криптография, RBAC, аудит действий.

---

## Глоссарий терминов для АИС КРР

Используй только эти термины в ТЗ:
- **ИС**: Информационная система.
- **ПС**: Подсистема (ПС-01, ПС-02 и т.д.).
- **КЗ**: Класс защищенности (для МО обычно КЗ-2 или КЗ-1).
- **СУБД**: Система управления базами данных (PostgreSQL 15).
- **ЭЦП / E-IMZO**: Система электронной цифровой подписи.

| Код Требования | Описание Функции | Приоритет | Метрика Успеха |
|----------------|------------------|-----------|----------------|
| REQ-01         | <Описание>       | High      | <Критерий>     |

### 2. Шаблон Раздела: Надежность и Производительность
"Система должна обеспечивать сохранность данных при аварийных ситуациях в электроснабжении. Время восстановления работоспособности после сбоя не должно превышать 2 часов. Резервное копирование базы данных должно выполняться ежедневно в 03:00."

### 3. Раздел: Эргономика и Интерфейс
"Интерфейс должен быть адаптирован для мониторов с разрешением от 1280x720 до 1920x1080. Цветовая гамма должна соответствовать брендбуку Министерства обороны. Навигация должна осуществляться не более чем в 3 клика до любой основной функции."

## Критерии Качества (MD Improver)
Каждое ТЗ должно быть:
- **Concise**: Плотное, человекочитаемое наполнение.
- **Actionable**: Требования должны быть проверяемыми (без слов "удобно", "красиво").
- **Current**: Информация должна отражать реальное состояние кодовой базы и API.

---

## Как проверить ТЗ

Когда пользователь дает текст для ТЗ, проверяй его по чек-листу:
1. [ ] Нет ли размытых формулировок типа "удобный интерфейс" (заменяй на метрики кликов).
2. [ ] Указаны ли требования к производительности?
3. [ ] Есть ли раздел про масштабируемость?
4. [ ] Соответствует ли структура стандарту 1987:2018?
