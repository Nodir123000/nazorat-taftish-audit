# Архитектура Справочных Данных (Reference Data Architecture)

## Основной Принцип: Справочники как Ядро

Справочники (классификаторы, словари) являются основой целостности данных приложения. Все сущности должны ссылаться на них, а не дублировать данные.

## Правила Моделирования Данных

### 1. Использование ID и Кодов для Связей

- **Всегда храните ID или Уникальный Код** (например, `unit_id`, `code="FIN"`, `code="GU_NI"`) в таблицах транзакций (Планы, Аудиты, Приказы).
- **НИКОГДА не храните отображаемый текст** (например, "Финансово-хозяйственная деятельность", "Главное управление...") напрямую в основных полях таблицы.
- Исключение: Исторические снимки данных (snapshots), которые не должны меняться при изменении справочника (например, название должности в подписанном документе).

### 2. Локализация и Отображение на Фронтенде

- Получайте ID/Код с бэкенда.
- Используйте утилиты локализации (например, `getLocalizedDirectionName`, `getLocalizedAuthorityName`) для преобразования Кода/ID в человекочитаемую строку в зависимости от текущей локали (`ru`, `uzLatn`, `uzCyrl`).
- Если утилита не находит значение по ID/Коду, это ошибка данных или утилиты, которую нужно исправить (добавить поиск по коду), а не "хардкодить" текст в базу.

### 3. Форматирование Дат

- **Храните даты как объекты `Date`** или строки в формате ISO-8601 (`YYYY-MM-DD`) в базе данных (`date`, `timestamp`).
- **Форматируйте даты для отображения** (например, "Январь 2024", "I квартал 2024") ТОЛЬКО на уровне UI/компонентов или в специальных вычисляемых полях (геттерах), но не в сырых данных БД.

## Алгоритм Работы

1. **Проверка Справочника**: Убедитесь, что значение существует в `lib/data/military-data.ts` или таблицах `ref_`.
2. **Сохранение (Seeding/Create)**: Сохраняйте только `id` или `code`.
3. **Отображение (Display)**: В компонентах используйте хелперы для рендеринга текста.
