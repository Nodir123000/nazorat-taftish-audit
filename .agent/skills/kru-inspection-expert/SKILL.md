# Эксперт по контрольно-ревизионной деятельности КРУ МО РУз

Этот навык содержит всю доменную базу знаний о КРУ (Контрольно-Ревизионное Управление) Министерства обороны Республики Узбекистан. Используй его для любых задач, связанных с предметной областью системы АИС КРР.

## Как использовать тяжелые справочники

Не загружай все файлы сразу. Используй только нужный:

### 1. Классификация нарушений (14 категорий)
**Файл:** `.agent/skills/kru-inspection-expert/reference/violations-categories.md`
Используй когда: пользователь описывает нарушение и нужно определить его тип, или при создании/редактировании формы нарушений.

### 2. Жизненный цикл ревизии
**Файл:** `.agent/skills/kru-inspection-expert/reference/audit-lifecycle.md`
Используй когда: вопросы о статусах ревизии, переходах между этапами, какие документы нужны на каком этапе.

### 3. Права пользователей (Матрица RBAC)
**Файл:** `.agent/skills/kru-inspection-expert/reference/user-roles-matrix.md`
Используй когда: нужно понять, кто что может делать в системе, или при реализации middleware и guard-ов.

### 4. Нормативная база
**Файл:** `.agent/skills/kru-inspection-expert/reference/legal-basis.md`
Используй когда: написание ТЗ, проверка соответствия требованиям стандартов, юридические формулировки.

### 5. Схема базы данных (краткий справочник)
**Файл:** `.agent/skills/kru-inspection-expert/reference/db-schema-quick-ref.md`
Используй когда: нужно быстро понять связи между таблицами, не читая весь schema.prisma.

## Ключевые бизнес-правила

1. **Ревизия без плана**: Внеплановые ревизии (`unplanned`) не привязаны к `rev_plan_year`, но обязательно имеют `order_id`.
2. **Комиссия**: Всегда должен быть ровно один `is_responsible = true` в `commission_members` для каждого приказа.
3. **Нарушение → Решение**: Каждое нарушение в статусе `registered` должно иметь хотя бы одно решение (`decisions`) в течение 30 дней.
4. **Сроки**: Исполнение решения помечается `overdue` автоматически через `deadline-watchdog` скилл.
5. **ПИНФЛ**: Поле из 14 цифр. Доступно только ролям `admin` и `chief_inspector`. Не выводить в публичных API.

## Структура данных: ключевые связи

```
rev_plan_year (Годовой план)
  └── orders (Приказы)
        └── commission_members (Члены комиссии)
        └── audits (Ревизии)
              └── violations (Нарушения)
                    └── decisions (Решения)
personnel / users (Инспекторы и личный состав)
```

## Как классифицировать нарушение (быстрый алгоритм)

1. Есть ли конкретная сумма ущерба в сумах? → **Финансовое** нарушение.
2. Связано с имуществом, техникой, инвентарем? → **Имущественное**.
3. Присвоение чужого? → **Хищение** (критическое).
4. Разница между документами и реальностью? → **Недостача**.
5. Переплата сверх норм? → **Незаконный расход** или **Переплата**.
6. Нарушение процедуры без прямого ущерба? → **Процедурное**.
